# 生成式推荐（GR）在讲什么？

生成式推荐将用户行为视作**类语言序列**，把「下一物品 / 下一组语义码」建模为**条件序列生成**问题：模型在语义 ID、离散 token 或结构化动作空间上自回归解码，而不仅是为候选集合逐对打分。它与大语言模型浪潮交汇，形成了 **LLM4Rec 探索 → 语义 ID 生成式检索 → 大规模序列骨干 → 端到端召排一体** 的工业演进路径。

以下内容压缩整理自站内调研材料（见项目 `GR-summary.html`），用于建立共同语境；**具体数字与业务结论请以原论文、技术报告及各家官方披露为准**。

## 一、传统级联架构为何触顶？

工业界长期采用「召回 → 粗排 → 精排 → 重排」漏斗：Wide&Deep、DIN、双塔等判别式模型把算力主要花在**候选集上的打分**。当业务成熟后，常见瓶颈包括：

1. **多阶段目标割裂**：各段独立优化，误差沿链路累积；模块间特征/缓存 IO 吃掉大量机器预算。
2. **语义与推理短板**：纯 ID 统计相关难以稳定处理长尾、冷启动、跨域意图。
3. **规模红利递减**：堆特征、加宽 MLP 的边际收益下降；需要新的**表示与生成**空间换取泛化。
4. **多目标权衡困难**：点击、时长、转化、多样性等目标在加权或帕累托方案下易「跷跷板」。

生成式路线尝试用**统一序列骨干 + 离散化物品空间**缓解上述矛盾，但并非所有场景都要「一夜端到端」。

## 二、四阶段演进（与上图节点对应）

### 1. 萌芽期（约 2015–2022）

- **序列生成**：GRU4Rec → SASRec → BERT4Rec，把「下一项」建模为序列任务，为后来的 token 化铺路。
- **VAE/GAN CF**：更多生成隐向量、缓解稀疏，而非直接生成最终候选列表。

### 2. LLM4Rec 探索期（约 2022–2023）

业界归纳出三条路线：

- **LLM Embeddings + RS**：文本/多模态语义进传统模型，落地门槛最低。
- **LLM Tokens + RS**：产出语义标签或中间 token，衔接召回。
- **LLM as RS**：端到端生成物品或指令结果——研究价值高，但词表规模、延迟与稳定性一度阻碍生产。

### 3. 爆发期（约 2023–2025）

- **TIGER / 语义 ID**：用 RQ-VAE 等把物品映射成短离散码，自回归解码＝生成式检索；解决「LLM 词表 vs 亿级 SKU」矛盾的重要思路。
- **HSTU（Meta）**：面向高基数流式序列的骨干，强调算力缩放与工程性能；公开论文给出大规模实验与线上结论描述。
- **OneRec（快手）**：Encoder–Decoder 统一召排、偏好对齐与成本优化并进，代表「端到端替换级联」的激进分支。

### 4. 深化落地期（约 2025 起）

- **混合架构成为默认选项之一**：在保留漏斗稳定性的前提下，用生成式模块增强召回/精排/重排（各家公开方案名称不一）。
- **排序骨干继续「卷算力效率」**：RankMixer、OneTrans 等强调**在延迟约束下规模化参数与特征交互**，并与序列建模统一。
- **垂直行业包**：电商（商品卡/直播）、广告等场景把生成式检索与业务约束（出价、GMV）绑定更紧。

## 三、三种范式（与「方案对比」页呼应）

| 范式 | 要解决的问题 | 典型代表方向 |
|------|----------------|--------------|
| **端到端生成（ID-as-Token）** | 全局一致性与架构简化 | 语义 ID + 自回归 Top-K；OneRec 系 |
| **融合式生成（模块增强）** | 风险可控、渐进替换 | TIGER 式召回、HSTU 式精排、LLM 重排 |
| **生成式内容与交互** | 体验与转化，不改主匹配 | 千人千面文案、推荐理由、对话导购 |

## 四、阅读与落地建议

- **先选瓶颈环节**：长尾召回、精排列表整体效用、还是展示层转化？再决定生成式介入点。
- **先验证约束**：P99 延迟、算力预算、内容安全与合规、可回放日志——再决定「端到端」还是「增强」。
- **把论文指标与线上指标分开**：离线 NDCG、公开技术报告中的 A/B 描述，需区分场景与流量定义。
